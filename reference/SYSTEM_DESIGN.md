# 🏗️ Confluence MGT Agent - System Architecture & Design

> 이 문서는 사내 Confluence 스페이스(SD -> AA 등)의 문서 자동화 및 자가 정화(Self-healing) 파이프라인 구축을 위한 핵심 아키텍처와 의사결정 사항을 기록한 마스터 문서입니다. 추후 다른 AI 에이전트나 새로운 개발자가 시스템을 유지보수할 때 참조하는 **단일 진실 공급원(SSOT)**입니다.

---

## 1. System Overview (시스템 개요)
이 시스템의 목적은 사용자들이 작성한 수많은 문서를 분석하여, 가장 적합한 폴더 구조에 배치하고 사내 정책에 맞는 레이블(태그)을 자동으로 부여하는 것입니다. 단순한 1회성 이관을 넘어, 스페이스 내부가 오염되지 않도록 스스로 정화하는 기능을 갖춥니다.

## 2. Core Architecture: 하이브리드 모델 (GitHub Actions + Dify)
시스템은 크게 '실행기(팔다리)'와 '두뇌'로 분리되어 작동합니다.
- **실행기 (GitHub Actions / Node.js 스크립트)**: 스케줄러(Cron) 및 수동 트리거로 동작하며, Confluence API를 호출해 문서를 수집/이동/업데이트합니다. 타임아웃 제한과 대용량 첨부파일 처리 제약을 극복합니다.
- **두뇌 (Dify Workflow / LLM)**: 수집된 문서의 제목과 본문을 받아, **"이 문서는 유효한가? 어느 폴더에 가야 하는가? 어떤 레이블이 붙어야 하는가?"**를 판별하여 JSON 형태로 실행기에게 지시를 내립니다.

## 3. Key Design Decisions (핵심 의사결정 사항)

### 3-1. 동적 컨텍스트 주입 (Dynamic Context Injection)
> *문제: Confluence의 폴더 구조(ID)가 변경/추가될 때마다 소스코드나 Dify 프롬프트를 수동으로 갱신해야 하는가?*

**결정: 스크립트 실행 시점에 실시간 폴더 구조를 수집하여 Dify에 주입한다.**
- 하드코딩된 폴더 ID 맵이나 지식베이스(RAG)를 사용하지 않습니다.
- 스크립트가 실행될 때마다 Confluence API를 호출해 **현재 AA 스페이스의 최신 구조**를 텍스트로 변환(`context_tree`)하여 Dify API 호출 시 입력 변수로 넘겨줍니다.
- 결과적으로 시스템은 완전한 자동화(Self-updating)를 달성합니다.

### 3-2. 폴더와 말단 문서의 구분 (`is-folder` 레이블 전략)
> *문제: Confluence는 폴더와 일반 페이지가 기술적으로 동일하다. LLM이 새로 들어온 문서를 '폴더'가 아닌 '기존 말단 문서' 하위로 배치하는 실수를 어떻게 막을 것인가?*

**결정: 컨테이너(폴더) 역할을 하는 페이지에만 전용 식별 레이블(`is-folder`)을 부착한다.**
- 스크립트가 동적 컨텍스트 트리를 수집할 때, Confluence 전체 페이지가 아닌 **`label="is-folder"`가 부착된 페이지들만** 가져와 트리를 구성합니다.
- LLM에게는 오직 '폴더(그릇)'들의 목록만 전달되므로, 말단 문서 밑으로 계층이 오염되는 것을 원천 차단합니다.
- 관리자는 새 폴더를 만들고 싶을 때, 페이지를 생성하고 `is-folder` 태그만 달아주면 됩니다.

### 3-3. 정책(Rule) 관리의 분리 (System Prompt vs RAG)
> *문제: 유효성 검증(Validation) 및 레이블(Label) 정책은 어디서 관리할 것인가?*

**결정: 명확성과 100% 준수를 위해 Dify 시스템 프롬프트(환경변수)에 텍스트로 명시한다.**
- 지식베이스(RAG) 문서를 활용하면 검색 실패로 인한 '환각(Hallucination)'이 발생할 위험이 있습니다.
- 정책 문서가 수십 페이지가 넘지 않는 이상, LLM의 시스템 프롬프트에 직접 기재하여 강제성을 높입니다.
- 유지보수를 위해 Dify의 Workflow 내 텍스트 변수(Text Variable) 블록으로 분리하여 비개발자도 쉽게 수정할 수 있게 합니다.

### 3-4. 노이즈 필터링 (Validation Logic)
> *문제: 데일리 스크럼, 개인 메모, 단순 회의록 같은 지엽적인 문서를 어떻게 걸러낼 것인가?*

**결정: LLM의 `is_valid` 판별을 통한 선제적 차단(Early Exit)**
- Dify 프롬프트 내에 **1. Validation Rule**을 명시합니다. (예: "일일 회의록이 아닐 것", "내용이 비어있지 않을 것", "2025년 이후의 데이터일 것" 등)
- LLM이 페이지 본문을 읽고 노이즈 문서라고 판단하면 JSON 응답으로 `{"is_valid": false}`를 반환합니다.
- GitHub Actions의 Migrator 스크립트는 이 값을 확인하고 `false`인 문서는 아예 복사(이관)를 시도하지 않고 즉시 스킵(Skip)합니다.
- 이를 통해 AA 스페이스에는 정말 가치 있는 문서(MPS, 기술조사, 프로젝트 현황 등)만 쌓이게 됩니다.

## 4. Workflows (운영 시나리오)

시스템은 다음 3가지 핵심 워크플로우를 지원합니다.

### 1) Migrator (외부 수집기)
- **트리거**: 주간 스케줄 (Cron)
- **로직**: SD 등 타겟 스페이스에서 최근 1주일간 수정된 문서를 수집 -> Dify에 판별 요청 -> AA 스페이스 내 적절한 `target_folder_id` 밑에 페이지를 생성하고 첨부파일 복사.

### 2) Auditor (내부 자가 정화기)
- **트리거**: 주간/월간 스케줄 (Cron)
- **로직**: AA 스페이스 내부에 존재하는 모든 말단 문서 스캔 -> Dify에 최신 정책 기준으로 재판별 요청 -> 현재 문서의 위치/레이블과 Dify의 판별 결과가 다를 경우, Confluence API를 통해 즉시 올바른 위치로 **이동(Move)** 및 레이블 업데이트.

### 3) Batch Utility (수동 일괄 처리기)
- **트리거**: 수동 (GitHub Actions Workflow Dispatch)
- **로직**: "기존 `group-ai` 태그를 `project-ai`로 전면 교체해라"와 같이, LLM의 지능적 판단이 불필요하고 명확한 1:1 치환이 필요할 때 사용. API만 호출하므로 매우 빠르고 비용(Token)이 발생하지 않음.

---

## 5. Exception Handling (Human-in-the-loop)
- LLM이 문서를 분석했을 때 주어진 `context_tree` 내에 도저히 적합한 폴더가 없다고 판단하면, 강제로 끼워 넣지 않습니다.
- 대신 JSON 응답에 `{"needs_new_category": true}`를 반환합니다.
- 스크립트는 해당 문서 처리를 보류(Skip)하고, 관리자에게 Slack/Email로 알림을 보냅니다. (예: "신규 분류가 필요합니다. 폴더를 생성하고 `is-folder` 태그를 달아주세요.")
